# Phase 4: 노트 기반 지식 그래프 (Knowledge Graph)
> 브랜치: `feature/issue-1048-knowledge-graph`  
> 이슈 번호: [#1048](https://github.com/jjaayy2222/flownote-mvp/issues/1048) | 마일스톤: `v9.0 - Adaptive Intelligence` | 상태: 🗓️ 예정

## 🎯 목표

기존의 단순 플랫 벡터 인덱스(FAISS 등) 기반 검색을 넘어서, 노트(문서) 간의 명시적 및 암묵적 관계를 **지식 그래프(Knowledge Graph)** 형태로 구축합니다. 
이를 통해 LangGraph 에이전트가 단편적인 텍스트 검색을 넘어 문맥과 연결망을 추론(`GraphRAG`)하도록 돕고, 프론트엔드에서는 시각화된 지식 맵을 통해 새로운 형태의 노트 탐색 경험을 제공합니다.

---

## 🏗️ 아키텍처 개요

```
[Markdown Notes]  →  [엔티티/릴레이션 추출 파이프라인]
                             ↓
[Graph DB / NetworkX] ← (노드: 노트/키워드, 엣지: 관계/참조)
                             ↓
[GraphRAG Router] ← (사용자 질의 → 벡터 검색 + 그래프 탐색 하이브리드)
                             ↓
[Next.js Frontend]
  └─ [react-force-graph] 지식 맵 뷰 시각화
  └─ [추천 모듈] 연결되지 않은 고립 노트 감지 및 제안
```

---

## 📋 세부 작업 (Tasks)

### 4-0. [공통] 지식 그래프 인프라 및 단일 진실 공급원(SSOT) 정책

> 💡 **참고 (경로 고정 규약)**: 본 문서에 명시된 주요 모델(Schema) 및 설정(Config)의 관리 주체와 책임 경계는 철저히 분리되며, 하드코딩 없는 SSOT 아키텍처를 유지합니다.

- [ ] **그래프 스키마 SSOT (`GraphNode`, `GraphEdge`)**
  - **소유권**: 백엔드의 스키마 정의 모델 (`backend/schemas/graph.py`)
  - **동기화**: 자동 생성된 OpenAPI 스키마를 통해 프론트엔드 타입 자동 생성. (프론트엔드 내 `Node`, `Link` 타입 수동 하드코딩 절대 금지)
- [ ] **설정 변수 SSOT (`GraphConfig`)**
  - **소유권**: `backend/core/config/graph.py`의 `GraphConfig` 클래스
  - **책임**: 그래프 탐색 깊이 제한(Max Depth), 노드 시각화 임계값, DB 연결 정보 등의 환경 변수를 캡슐화

### 4-1. 그래프 노드/엣지 생성 및 관리 로직

- [ ] **그래프 백엔드 엔진 선택 및 연동 (ADR 명세)**
  - 데이터 규모 임계값(Total Nodes < 5,000) 및 초기 배포 환경(단일 유저 중심의 로컬 개인화)을 고려하여 **일차적으로 `networkx`를 채택**.
  - `networkx` 사용 시, 파일 시스템 기반 저장 경로(`STORAGE_BASE_PATH`)에 해시된 사용자 ID를 적용하여 데이터 프라이버시 및 테넌트 격리를 보장함.
  - 향후 다중 사용자 간 거대 지식망 공유, 동시성 락(Lock) 처리, 분산 환경이 요구될 경우 `neo4j`로 무중단 마이그레이션이 가능하도록 **Repository 추상화 패턴** 적용 필수.
- [ ] **엔티티/릴레이션 자동 추출 파이프라인**
  - 노트 내용 내의 위키링크(`[[노트명]]`)나 태그를 파싱하여 명시적 엣지(Explicit Edge) 생성.
  - LLM 또는 NLP 기법을 이용해 키워드 기반의 암묵적 엣지(Implicit Edge) 가중치 생성.

### 4-2. GraphRAG 하이브리드 라우터 구현

- [ ] **에이전트 그래프 통합 로직 (`backend/agent/graph_router.py`) 신설**
  - 기존 벡터 검색(Vector RAG) 결과와 지식 그래프 탐색 결과를 결합하는 하이브리드 쿼리 라우터 개발.
  - **방어적 코딩 명세**: 그래프 순회 시 무한 루프에 빠지지 않도록 환경 변수(`GRAPH_MAX_TRAVERSAL_DEPTH`)로 최대 깊이를 강제(Clamping 적용).
  - 지식 그래프에서 얻은 연결 컨텍스트를 프롬프트에 주입하여 에이전트 답변의 해상도 향상.

### 4-3. 프론트엔드 그래프 시각화

- [ ] **지식 맵 뷰 컴포넌트 (`frontend/src/components/GraphView`) 신설**
  - 라이브러리: `react-force-graph` (2D/3D 지원 여부 확인 후 적용).
  - 대량의 노드 렌더링 시 브라우저 OOM(Out of Memory)이나 프리징 방지를 위한 렌더링 최적화.
    - 성능 병목 발생 시, 표시 노드 수를 환경 변수(`NEXT_PUBLIC_MAX_GRAPH_NODES`)에 맞춰 동적으로 제한.
  - 노드 클릭 시 해당 노트 내용 프리뷰 또는 라우팅 이동 지원.

### 4-4. 고립 노트 감지 및 추천 기능

- [ ] **고립 노트 감지 백그라운드 스케줄러 (연결성 분석 알고리즘)**
  - 그래프 차수가 `0` 또는 극도로 낮은 '고립 노트(Orphan Notes)' 식별 프로세스 구축.
  - **스케줄러 아키텍처**: 운영 가시성과 배포 독립성을 위해 앱 내부의 단순 `BackgroundTasks`를 지양하고, **Celery 기반 비동기 워커** 또는 **AWS EventBridge + Lambda** (또는 K8s CronJob) 형태의 독립 배치 잡으로 설계.
  - **보안 및 격리**: 전역 스캔 시 다른 사용자의 노트 컨텍스트가 섞이지 않도록(Data Leakage 방지), 쿼리 실행 전 반드시 `hashed_user_id` 기반 테넌트 격리 컨텍스트 적용 보장.
  - 벡터 유사도(Vector Similarity)가 높지만 명시적 엣지가 없는 노드 쌍을 추출하여 사용자에게 "연결(Link) 추천" 알림 제공.

### 4-5. [공통] 설정 로딩 및 검증 파이프라인 (Validation)

- [ ] **설정 책임 분리와 검증 레이어 강제**
  - **[선언부] `backend/core/config/graph.py` (`GraphConfig`)**: 환경 변수 모델링, 기본값 정의, 상수 범위 선언만 담당 (실제 로딩 수행 금지).
  - **[검증부] `backend/core/config_validator.py` (`GraphValidator`)**: `GraphConfig`의 상수를 참조하여 애플리케이션 시작(Startup) 시점에 OS 환경 변수 로드 수행.
    - 범위 이탈 시 즉각적인 **Clamping 보정 로직** 적용 및 `WARNING` 구조화 로그 출력.
    - 치명적인 로딩 실패 시 서브시스템(Subsystem) 단위를 `DEGRADED`로 격하하여 전체 부팅 중단 방지 (Subsystem Fail-fast 정책 적용). 하드코딩 절대 금지.

  | 환경 변수 | 타입 | 기본값 | 유효 범위 | 파싱 오류 시 동작 |
  |-----------|------|--------|-----------|------------------|
  | `GRAPH_MAX_TRAVERSAL_DEPTH` | int | 3 | 1~5 | Clamp + WARNING |
  | `NEXT_PUBLIC_MAX_GRAPH_NODES` | int | 500 | 50~2000 | Clamp + WARNING |
  | `GRAPH_DB_URL` | str | "" | - | 비어있을 시 networkx Fallback |

## ✅ 완료 기준 (Definition of Done)

- [ ] 위키링크 및 태그 파싱을 통해 노트 간 엣지가 성공적으로 생성 및 저장됨.
- [ ] LangGraph 에이전트가 쿼리 시, 그래프 인접 노드의 컨텍스트를 포함하여 답변을 생성할 수 있음.
- [ ] 프론트엔드에서 `react-force-graph` 기반 시각화 맵이 정상적으로 로드 및 상호작용 가능함.
- [ ] 연결되지 않은 고립 노트에 대한 LLM 기반 연결 추천 기능이 작동함.
- [ ] 그래프 탐색 한계, 시각화 최대 노드 수 등이 환경 변수로 제어되며 범위를 벗어날 경우 Clamping이 작동함.
