# Phase 3: 실시간 스트리밍 & 프론트엔드 UX (Realtime Streaming)
> 브랜치: `feature/issue-1047-realtime-streaming`  
> 이슈 번호: [#1047](https://github.com/jjaayy2222/flownote-mvp/issues/1047) | 마일스톤: `v9.0 - Adaptive Intelligence` | 상태: 🗓️ 예정

## 🎯 목표

현재 LangGraph 에이전트의 응답 방식은 전체 생성이 완료된 후 단일 블록으로 반환되어  
사용자가 답변을 기다리는 체감 시간(Perceived Latency)이 큼.  
Server-Sent Events(SSE) 기반 스트리밍 파이프라인을 구축하고  
Next.js 프론트엔드를 실시간 렌더링 UX로 전환하여  
**사용자 체감 응답 속도를 극적으로 개선**하고 대화형 AI 경험을 완성합니다.

---

## 🏗️ 아키텍처 개요

```
[LangGraph Agent]
  └─ astream_events() / AsyncIterator<토큰 청크>
        ↓
[FastAPI SSE Endpoint]  ← POST /api/chat/stream
  └─ StreamingResponse
        ↓  HTTP/1.1  text/event-stream
[Next.js Frontend]
  └─ fetch API (ReadableStream)
        ↓
[UI: 실시간 렌더링]
  ├─ 타이핑 인디케이터 (생성 중)
  ├─ 마크다운 스트리밍 파싱
  └─ 코드 블록 하이라이팅 (완료 시)
```

    - `TokenChunk`: `{ "type": "token", "data": string }`

### 3-2. 스트리밍 API 엔드포인트 구현

- [ ] 스트리밍 전용 API 라우터 신설 (예: `POST /api/chat/stream`)
  - **인증 필수**: 기존 인증 미들웨어 재사용 (`hashed_user_id` 내부 전달, `user_id` 원문 로그 금지)
  - **요청 바디**: 기존 비스트리밍 요청 스키마 공유

  - **SSE(Server-Sent Events) 응답 표준 구현**:
    - 생태계 친화적인 검증된 SSE 라이브러리를 채택하여 스트리밍 응답 반환
    - 특정 라이브러리에 과도하게 종속되지 않도록 응답 생성 래퍼(Wrapper) 패턴 적용 권장
    - `Content-Type: text/event-stream` 헤더 설정
    - `Cache-Control: no-cache`, `X-Accel-Buffering: no` 헤더 명시 (리버스 프록시 버퍼링 방지)

  - **연결 수명 주기 관리**:
    - 클라이언트 연결 종료(Disconnect) 감지: `request.is_disconnected()` 폴링 또는 `asyncio.CancelledError` 감지
    - 연결 종료 시 LangGraph 에이전트 스트리밍 코루틴을 즉시 정리하여 리소스 누수 방지
    - 백프레셔(Backpressure) 제어: 큐(asyncio.Queue) 기반 토큰 버퍼를 통해 프로듀서-컨슈머 속도 불일치 흡수

  - **설정 외부화** (하드코딩 금지):
    - `SSE_KEEPALIVE_INTERVAL_SECS` [int, 기본값: 15, 범위: 5~60]: SSE keepalive 핑 간격
    - `STREAM_BUFFER_MAX_SIZE` [int, 기본값: 100, 범위: 10~1000]: 토큰 큐 최대 크기
    - `STREAM_TIMEOUT_SECS` [int, 기본값: 120, 범위: 30~600]: 스트리밍 세션 최대 허용 시간

  - **관측성(Observability)**:
    - 스트리밍 시작/완료/에러를 `[STREAM]` 태그 구조화 로그로 기록
    - TTFT(Time To First Token) 측정: 요청 수신 ~ 첫 토큰 발행 사이의 latency를 밀리초 단위로 기록
    - 총 스트리밍 시간 및 발행된 토큰 수(청크 수) 메트릭 기록

- [ ] 기존 `/api/chat` 비스트리밍 엔드포인트 유지:
  - 스트리밍이 불가한 클라이언트(구형 브라우저, CLI 도구 등)를 위해 기존 동기 응답 엔드포인트는 제거하지 않고 병행 운영
  - 두 엔드포인트 간 비즈니스 로직 중복 방지: 공통 에이전트 실행 로직은 헬퍼 함수로 분리하여 재사용

- [ ] 라우터 등록:
  - `backend/api/endpoints/__init__.py` 및 `backend/api/routes.py`에 `chat_stream_router` 등록

#### 세부 작업 순서

### 3-5. [공통] 스트리밍 설정 검증 및 단일 진실 공급원(SSOT) 정책

- [ ] 설정 변수의 중앙 집중식 SSOT 모듈 구축
  - **SSOT(Single Source of Truth) 원칙**: 백엔드와 프론트엔드 간 타임아웃, 버퍼 사이즈 등 설정 파편화를 막기 위해 공유 설정 모듈(예: 백엔드의 중앙 Config 클래스)에 기본값과 클램핑(Clamping) 규칙을 단일 정의.
  - 문서와 실제 코드 간의 괴리(Drift)를 방지하기 위해 개별 모듈 하드코딩 금지.

- [ ] 스트리밍 관련 환경 변수에 대한 유효성 검사 구성
  - 기존 전역 가용성 상태 레지스트리(Subsystem)에 스트리밍 모듈 추가
  - **장애 전파 범위 원칙** (Phase 2 정책과 동일하게 적용):
    - 설정 파싱 오류 시 침묵적 폴백(Silent Fallback) 금지
    - 전체 서비스 중단 대신 스트리밍 서브시스템만 비활성화하고, 기존 비스트리밍 엔드포인트 생존성 유지 (Subsystem Hard Failure)
    - 헬스 체크 시스템에 `DEGRADED` 상태 노출로 운영 가시성 확보

  - **환경 변수 목록 및 검증 규칙**:

    | 환경 변수 | 타입 | 기본값 | 유효 범위 | 파싱 오류 시 동작 |
    |-----------|------|--------|-----------|------------------|
    | `SSE_KEEPALIVE_INTERVAL_SECS` | int | 15 | 5~60 | Clamp + WARNING |
    | `STREAM_BUFFER_MAX_SIZE` | int | 100 | 10~1000 | Clamp + WARNING |
    | `STREAM_TIMEOUT_SECS` | int | 120 | 30~600 | Clamp + WARNING |
    | `LANGGRAPH_STREAM_VERSION` | str | "v2" | "v1" / "v2" | 기본값 폴백 + WARNING |
    | `NEXT_PUBLIC_USE_STREAMING` | bool | true | true / false | 기본값 폴백 + WARNING |

  - **프론트엔드 환경 변수**는 Next.js `NEXT_PUBLIC_` 접두사 규칙에 따라 빌드 타임 주입
    - 스트리밍 피처 플래그(`NEXT_PUBLIC_USE_STREAMING`)로 점진적 롤아웃 및 긴급 비활성화 지원
