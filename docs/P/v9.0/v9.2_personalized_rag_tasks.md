# Phase 2: 사용자 개인화 RAG (Personalized RAG)
> 브랜치: `feature/issue-1046-personalized-rag`  
> 이슈 번호: [#1046](https://github.com/jjaayy2222/flownote-mvp/issues/1046) | 마일스톤: `v9.0 - Adaptive Intelligence` | 상태: 🗓️ 예정

## 🎯 목표
현재 모든 사용자에게 동일하게 제공되는 전역 FAISS 인덱스 기반 RAG 시스템에  
사용자별 상호작용 패턴 분석 및 개인화된 컨텍스트 창(Personalized Context Window)을 결합하여  
사용자 맞춤형 AI 응답 품질을 획기적으로 향상시킵니다.

---

## 📋 세부 작업 (Tasks)

### 2-1. 사용자별 임베딩 서브-인덱스 구축 및 관리
- [ ] `backend/services/personalized_index_service.py` 신설
- [ ] 사용자별 임베딩 서브-인덱스 생성 및 저장
  - **Path Strategy**: `${STORAGE_BASE_PATH}/user_data/idx/{hashed_user_id}.faiss`
  - **Env Examples**:
    - 로컬 개발: `STORAGE_BASE_PATH=/Users/jay/data/v9`
    - 운영 환경: `STORAGE_BASE_PATH=/mnt/efs/v9/prod_root`
  - 사용자 ID는 파일 시스템에 직접 노출하지 않도록 해시값(`hashed_user_id`)으로 경로 생성
    - **해시 전략**: `SHA-256(user_id + per_user_salt + global_pepper)` 구조를 사용하여 결정론적 경로 생성을 보장하되 무차별 대입(Brute-force) 역추적에 강인하도록 설계
      - `per_user_salt`: DB의 사용자 레코드에 일반 평문으로 저장 (보안성보다는 고유성 목적)
      - `global_pepper`: 애플리케이션 시작 시 AWS KMS 또는 HashiCorp Vault와 같은 시크릿 매니저를 통해 메모리에만 안전하게 주입 (소스 코드나 `.env` 평문 저장은 엄격히 금지)
        - **장애 조치(Edge Case)**: 애플리케이션 부트스트랩 또는 정기 키 로테이션 중 KMS/Vault 조회 실패 시 예외 원인별 동작 분리 지원:
          - 일시적 네트워크 장애(`TimeoutException`, `ConnectionError`): 타임아웃 5초, 최대 3회 백오프 재시도 수행
          - 치명적 권한/존재 오류(`AccessDeniedException`, `NotFound` 등): 재시도 없이 즉시 프로세스를 중단(Hard Fail-fast)하여 개인정보 레코드의 암호화 무단 오염 및 노출 원천 차단
- [ ] 인덱스 수명 주기 관리: 신규 생성 / 증분 업데이트(Incremental Update) / 주기적 재구축(Compaction) 전략 수립
  - **컴팩션 트리거 예시**: 서브-인덱스 내 누적 벡터 개수가 500개를 초과하거나, 잔여 벡터 대비 삭제된 벡터의 비율(`deleted/total`)이 15%를 초과할 경우 백그라운드 워커에서 인덱스 재생성(Rebuild) 수행
    - 다양한 워크로드 확장에 원활하게 대비하기 위해 해당 임계값은 환경 변수(`FAISS_COMPACTION_VECTOR_THRESHOLD` [int, 기본값: 500, 최소값: 100], `FAISS_COMPACTION_DELETE_RATIO` [float, 기본값: 0.15, 유효 범위: 0.0~1.0])로 외부화하여 동적 튜닝이 가능하도록 설계
      - *Validation Policy*: 파싱 불가능한 값(non-parsable)이거나 유효 범위를 벗어난 값이 주입될 경우, 예기치 않은 동작(Silent Fallback) 방지를 위해 애플리케이션 시작 단계에서 즉시 중단 예외(Hard Failure) 발생
- [ ] 인덱스 메타데이터(생성일, 최종 업데이트, 벡터 수) Redis 캐시 저장

### 2-2. 사용자 관심 토픽 클러스터링
- [ ] 사용자 검색 히스토리 기반 관심 토픽 자동 추출 (`backend/services/topic_clustering_service.py`)
- [ ] k-means 알고리즘을 활용한 토픽 클러스터링 구현
  - 클러스터 수(k) 결정 로직: 엘보우 메서드(Elbow Method) 또는 사일루엣 계수(Silhouette Score)
- [ ] 클러스터링 결과를 Redis에 캐싱하여 반복 연산 방지 (TTL: 환경 변수 `TOPIC_CLUSTER_CACHE_TTL` 기반)
- [ ] 히스토리 데이터 부족(Cold Start) 사용자를 위한 전역 인덱스 폴백(Fallback) 전략
  - **정량적 기준**: 완료된 챗봇 질문/응답(메시지 쌍) 횟수 또는 유효한 RAG 검색(Search) 수행 기록이 누적 5건 미만인 사용자에 한정
  - **기본 결정 규칙(Default Rule)**: 해당 사용자의 개인화 가중치를 무효화하고 무조건 전역 인덱스 100%(`GLOBAL_INDEX_WEIGHT=1.0`) 폴백으로 처리하여 품질 저하 방지

### 2-3. 개인화 컨텍스트 가중치 라우팅
- [ ] `backend/services/hybrid_search_service.py` 확장: 개인 인덱스(60%) + 전역 인덱스(40%) 혼합 검색 라우터 구현
  - 추천 혼합 비율은 환경 변수 `PERSONALIZED_INDEX_WEIGHT` [float, 기본값: 0.6, 유효 범위: 0.0~1.0] 및 `GLOBAL_INDEX_WEIGHT` [float, 기본값: 0.4, 유효 범위: 0.0~1.0]로 오버라이드 가능 (단, 콜드 스타트 조건 부합 시 즉시 전역 100% 편향 로직 적용)
    - *Validation Policy*: 파싱 불가능한 값(non-parsable)이거나 유효 범위를 벗어난 값이 주입될 경우, 예기치 않은 동작(Silent Fallback) 방지를 위해 애플리케이션 시작 단계에서 즉시 중단 예외(Hard Failure) 발생
- [ ] LangGraph 에이전트 `retrieve_node`에서 개인화 서브-인덱스 우선 참조하도록 수정
- [ ] 개인 인덱스 검색 결과와 전역 인덱스 결과 간의 RRF(Reciprocal Rank Fusion) 병합 로직 구현

### 2-4. 개인정보 보호(Privacy) 및 GDPR 준수
- [ ] 사용자 데이터 접근 감사 로그(Audit Log) 생성 (`[OBS]` 태그 준수)
- [ ] 사용자 삭제 요청(Right to Erasure) 처리 파이프라인 구축:
  - **Vector Data 삭제**:
    - DB 레코드 물리적 삭제 (`SQL DELETE`)
    - 매일 02:00 AM 또는 누적 1,000건 이상 삭제 시 정기 `VACUUM` 수행
    - FAISS 인덱스 영구 제거 및 주기적 재구축(Compaction)으로 물리 데이터 잔존 리스크 차단
  - **Interaction Log 익명화**:
    - **알고리즘**: PBKDF2-HMAC-SHA256
      - `PBKDF2_ITERATIONS` 환경 변수를 통해 프로덕션 하드웨어 및 트래픽 패턴에 맞게 동적 튜닝 가능하도록 설계
      - **초기 기본값 설정 배경**: 600,000회 (AWS t3.medium / Intel Xeon Platinum 동급 환경에서 Single-threaded 연산 기준 타겟 지연 시간 0.3s 이내 결과를 도출한 권장값)
    - **키 전략**: Global Pepper + Per-User Salt 조합 / 연 1회 키 로테이션(Annual Rollover)
      - Pepper 키의 안전성을 위해 외부 KMS 볼트(Vault)와 연동하여 런타임에 메모리로만 참조
      - N-1 버전 키는 암호학적 과도기 처리를 위해 최대 1년 보관 후 자동 폐기
      - N-2 이하 구형 키 보유 휴면 데이터는 전체 시스템 수명 주기 정책에 따라 일괄 삭제
  - **준수 기준**: NIST SP 800-132 (PBKDF2 표준) 강제 준수 — 암호학적 엔트로피 결합으로 실질적 재식별 리스크 최소화
  - **Audit Logging**: 삭제 처리 결과 감사 로그 90일 보관 후 자동 영구 삭제 스케줄러 가동
